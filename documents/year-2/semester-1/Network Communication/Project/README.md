# Network Communication Project

## Overview

This multi-task group project covers building and network design, feasibility, implementation ideas, configuration evidence, and final reporting.

## Objectives

- Design a network for the assigned environment.
- Evaluate feasibility and document technical choices.
- Configure, test, and present the proposed solution.

## Tools and Technologies

Network design, Cisco Packet Tracer-related work, document authoring, and diagrams.

## What I Did

I contributed to the group network design, documentation, configuration evidence, and task submissions.

To complete this project, I focused on planning, configuring, testing, and documenting the relevant part of the network solution. I reviewed the available files and outputs together so that the implementation, written explanation, and supporting evidence remained connected to the task objectives. ## Key Learning Outcomes

I learned how technical network choices must respond to building, user, service, cost, and reliability requirements.

The work also strengthened my understanding of network topology, addressing, connectivity, device configuration, testing, and technical documentation. I became more aware that a technical result should be supported by a clear process, suitable evidence, and an explanation of why the selected approach was appropriate. This will help me approach related tasks more systematically in future coursework.

## Reflection

The staged tasks made the project easier to manage, but consistency between task documents was an ongoing challenge.

The most important challenge was identifying whether a connectivity problem came from the physical design, addressing plan, or device configuration. Working through this required me to check the task in smaller stages and compare the result with the available requirements or evidence. This process improved my patience and made me more careful about verifying technical work before presenting it.

Overall, this project contributed to my development by combining practical work with documentation and reflection. It showed me which parts I can now complete with more confidence and which areas still require further practice. The experience will be useful when I work on larger projects that demand clearer organisation, stronger testing, and more independent technical decisions.

## Possible Improvements

A useful next step would be to add one final architecture diagram and addressing plan. A useful next step would be to identify final versions of duplicated documents.

In a future version, I would also focus on labelled topology diagrams, configuration tables, test results, and a clearer record of troubleshooting decisions. These additions would make the work easier to understand, reproduce, and evaluate while providing stronger evidence of the development process.
